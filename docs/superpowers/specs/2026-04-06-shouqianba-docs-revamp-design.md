# 收钱吧开发者文档站改版设计

日期：2026-04-06

## 1. 背景

本次改版面向收钱吧开发者文档站，目标不是先做全面内容治理，而是优先解决新开发者首次进入站点时“不会判断接入路线、找不到第一步该做什么”的问题。

本轮脑暴过程中已经确认：

- 主要服务对象是服务商 / ISV 技术团队，不以商户自研团队为第一优先。
- 首页优先解决“第一次接入太难”，而不是“现有内容结构太乱不好找”。
- 首页第一步应让用户先按接入场景 / 产品线选择集成方式，而不是先看接口目录。
- 首页整体结构采用“接入导航型首页”。
- 视觉系统只参考收钱吧官网的颜色关系与按钮、卡片质感，不改动已确认的信息架构。

## 2. 设计目标

### 2.1 核心目标

- 让首次进入的服务商 / ISV 技术团队可以在 30 秒内判断自己的接入路线。
- 让用户先进入对应 Quickstart，再逐步进入 API 参考、验收文档、FAQ、最佳实践。
- 把首页从“文档目录页”改造成“接入决策页”。

### 2.2 成功标准

- 新用户进入首页后可以快速理解站点不是按接口堆砌，而是按接入路径组织。
- 用户可以在 3 次点击内进入正确的主路线与对应 Quickstart。
- 搜索能优先命中场景词和方案页，而不是只返回底层 API 条目。

### 2.3 非目标

- 本轮不以全面重写所有历史文档为目标。
- 本轮不把开发者站改造成官网式营销门户。
- 本轮不以行业方案作为首页第一层主骨架。

## 3. 参考策略

### 3.1 外部参考

- 信息架构参考 Stripe 文档站：先按能力与接入路线组织，再进入深层文档。
- onboarding 体验参考 OpenAI Docs：强化首屏引导、快速开始和搜索入口。
- 视觉颜色与控件手感参考收钱吧官网：使用更贴近官网的暖色品牌关系与更柔和的按钮、卡片质感。

### 3.2 最终组合

本次改版采用如下组合原则：

- Stripe 式接入架构
- OpenAI 式 onboarding
- 收钱吧官网式颜色和控件质感

## 4. 现有内容资产提炼

当前工作目录中的内容不是站点源码，而是一组场景资料、最佳实践、FAQ 与验收文档。这些材料显示现有内容主要按“接入场景 / 终端形态 / 扩展能力”组织，而不是按“行业”组织。

从现有文件中提炼出的主要内容簇如下：

- 线下门店收款：B扫C、C扫B-PRO、C扫B-预下单、刷脸支付
- 线上与移动端支付：App 支付、公众号支付、手机浏览器 H5、微信/支付宝内 H5
- 小程序生态支付：小程序支付、小程序插件、小程序 B2B 支付、小程序微企付
- 平台型能力：收付通开户、收付通交易
- 扩展能力：交易事件通知、支付网关回调签名、智慧门店开放平台

说明：

- “微信/支付宝内打开 H5 支付”在资料中已经带有“不推荐”信号，因此不应作为首页主推荐路线。
- 首页主骨架如果按餐饮、零售等行业组织，会与当前资产沉淀方式错位。

## 5. 用户与核心任务

### 5.1 目标用户

首页默认服务对象是服务商 / ISV 技术团队，典型角色包括：

- 新接入的技术负责人
- 首次联调的研发工程师
- 临近上线、需要验收与排障的实施或交付同学

### 5.2 核心任务

首次来到站点时，这类用户最需要完成的动作是：

1. 判断自己属于哪条接入路线
2. 找到最小可跑通路径
3. 再进入 API、验收、FAQ 和最佳实践

## 6. 信息架构

### 6.1 首页角色

首页不是传统文档封面，也不是 API 索引页，而是一个接入导航页。它的首要职责是帮助用户完成接入路线选择。

### 6.2 首页第一层主分流

首页第一层固定为以下 5 类主分流：

1. 线下门店收款
2. 线上与移动端支付
3. 小程序支付
4. 商户入网与分账
5. 开放能力与通知

### 6.3 各主分流对应子场景

#### 线下门店收款

- B扫C
- C扫B-PRO
- C扫B-预下单
- 刷脸支付

#### 线上与移动端支付

- App 支付
- 公众号支付
- 手机浏览器 H5

说明：微信/支付宝内 H5 不作为主推荐入口，仅保留兼容说明或迁移说明。

#### 小程序支付

- 小程序支付
- 小程序插件
- 小程序 B2B 支付
- 小程序微企付

#### 商户入网与分账

- 收付通开户
- 收付通交易

#### 开放能力与通知

- 交易事件通知
- 支付网关回调签名
- 智慧门店开放平台

## 7. 页面模块设计

### 7.1 顶部导航

顶部导航服务熟手直达，不承担新手分流任务。建议放置：

- 接入方案
- API 参考
- 最佳实践
- 更新日志
- 支持与帮助

### 7.2 首屏 Hero

首屏只回答一个问题：你应该从哪条路线开始接入。

首屏组成：

- 面向服务商 / ISV 的定位文案
- 主 CTA：开始接入
- 辅助 CTA：查看 5 条主路线
- 搜索入口：支持搜场景词、API 名、FAQ

文案语气应偏“接入引导”，而不是“接口说明书”或“官网营销页”。

### 7.3 主分流区

第二屏直接展示 5 张主路线卡片。每张卡片只承载：

- 主路线名称
- 一句适用场景说明
- 2 到 3 个代表性子场景词

首页不在此区堆叠过多文档细节，重点是帮助快速判断路线。

### 7.4 Quickstart 路径说明区

第三屏用于解释每条路线进入后会遵循的统一结构：

1. 场景判断
2. 接入准备
3. 最小可跑通链路
4. 验收与上线

这一区域的作用是提前建立用户预期，让用户知道点击后不是直接掉进接口目录。

### 7.5 常用直达入口

第四屏提供高频直达入口，服务已经知道自己要什么的用户：

- API 参考
- 验收文档
- FAQ
- 收付通
- 事件通知
- 回调签名

### 7.6 页尾

页尾承接长期维护信息：

- 更新日志
- 状态 / 公告
- 支持渠道

## 8. 用户路径设计

### 8.1 新 ISV 技术同学

首页 -> 选择主路线 -> 进入对应 Quickstart -> 跑通第一笔交易 -> 查看 API 参考与验收文档

### 8.2 有明确目标的熟手

首页 -> 使用搜索或顶部导航 -> 进入方案页、API 页或支持文档

### 8.3 临近上线的实施同学

首页 -> 直达验收文档、FAQ、回调签名、事件通知 -> 完成联调与排障

## 9. Quickstart 设计规范

每条主路线下的 Quickstart 都应遵循统一框架：

1. 我适合这个方案吗
2. 接入前准备什么
3. 第一笔交易最小链路
4. 常见错误与排查
5. 验收与上线检查项
6. 深入阅读入口

深入阅读入口统一指向：

- API 参考
- 验收文档
- FAQ
- 最佳实践
- 更新日志

## 10. 搜索策略

搜索不应只按接口名工作，而要优先支持场景词搜索。

推荐优先级：

1. 方案页 / Quickstart
2. 相关 API
3. FAQ / 验收文档 / 最佳实践

应优先支持的搜索词包括：

- B扫C
- C扫B
- App 支付
- 公众号支付
- 小程序插件
- 收付通开户
- 回调签名
- 事件通知

## 11. 错误处理与内容约束

### 11.1 选错路线时的纠偏

每条方案页都应包含“你可能想找的是”一类的跨路线引导，降低首次选择错误带来的成本。

### 11.2 历史或不推荐方案处理

历史兼容方案不从首页主推荐入口进入，仅在相关方案页保留：

- 适用前提
- 限制说明
- 替代方案

### 11.3 行业内容的处理方式

行业方案可以作为辅助入口或推荐卡片出现，但不作为首页第一层主骨架。这样既能保留官网的业务感，也不会与现有内容结构冲突。

## 12. 视觉与品牌应用规则

### 12.1 不变的部分

- 不改变接入导航型首页的信息架构
- 不改变 5 类主分流骨架
- 不把首页改造成官网式营销页

### 12.2 借鉴收钱吧官网的部分

- 暖橙、米白、深棕灰为核心色彩关系
- 按钮与卡片更圆润，质感更柔和
- 控件 hover 与选中态使用品牌暖色
- 局部使用轻量暖色渐变增强品牌识别

### 12.3 需要克制的部分

- 不把官网销售型文案搬进开发者站
- 不让“申请开通”类 CTA 压过“开始接入”
- 不让品牌色大量侵入正文、表格、代码块和技术导航

### 12.4 最终原则

结构像现代开发者站，颜色和控件手感像收钱吧官网。

## 13. 验证方式

改版成功与否主要看接入效率，而不是视觉是否更炫。

建议至少验证以下场景：

1. 一个完全陌生的 ISV 技术同学能否在 30 秒内选出正确主路线
2. 一个已知要接公众号支付的用户能否在 3 次点击内进入正确方案
3. 一个准备上线的同学能否快速找到验收文档和回调签名说明

## 14. 实施边界

本设计文档只定义改版方向、信息架构、视觉边界和用户路径，不包含具体技术实现方案。实现阶段应再单独拆解：

- 首页与导航改版
- 方案页模板与 Quickstart 模板
- 搜索结果优先级调整
- 内容迁移与重组策略

## 15. 结论

本次收钱吧开发者文档站改版，应以“新 ISV 首次接入更容易”为第一目标，通过“接入导航型首页 + 5 类主分流 + Quickstart 优先”的结构，重建首页与方案页的引导能力。

视觉上不改变已确认信息架构，只把颜色关系与按钮、卡片质感向收钱吧官网靠拢。最终形态应是一个更现代、更友好、但仍然高效率的开发者文档站，而不是一个营销化官网页面。
